# 14 — Work and Projects

`04-work/` for paid engagements. `05-projects/` for personal project lifecycle. Two folders, two rituals, one rule about confidentiality.

> Last updated: 2026-06-14. The live source of truth is `04-work/CLAUDE.md` and `05-projects/CLAUDE.md`. This chapter explains; those files govern.

## Two folders, two purposes

| Folder | Purpose | Lives outside vault |
|--------|---------|---------------------|
| `04-work/` | Paid engagements + ISO-week plans | (no) |
| `05-projects/` | Personal projects (planning + active + completed) | Code lives in `~/projects/` |

Some projects are both (a paid client engagement that is also a personal project). They go in `04-work/` if the primary frame is "paid engagement" and in `05-projects/` if the primary frame is "thing I am building." Overlap is allowed and explicit — see Helios below, which has an engagement folder in `04-work/helios/` AND a completed retrospective in `05-projects/completed/helios/`.

#### /work weekly-planner

Run on Monday morning (or the first work session of the week).

Reads:
- `02-ana/identity/goals.md`
- Last week's plan
- Open work-related items across `04-work/`

Writes:
- `04-work/work-plans/{YYYY-WNN}.md`

Template sections: Non-negotiables / By engagement (one section per `04-work/` folder) / Personal-skills / Review cadence. The output is concise. Style preference: "Plans should be extremely concise. Sacrifice grammar for concision." `allowed-tools`: Read, Write, Bash, AskUserQuestion.

#### /work meeting-prep

Run before a meeting.

Reads:
- The relevant engagement's recent meetings and context docs
- The engagement's proposals and diagrams
- This week's work-plan

Writes:
- A briefing INSIDE the engagement folder: `04-work/{engagement}/meeting-{YYYY-MM-DD}-{topic-slug}.md`.

Sections: Attendees / Objective / Context / Proposed agenda / Questions to ask / Decisions needed / Risks-watch-outs / Follow-ups. Briefings live inside the engagement folder, not at the `04-work/` root. `allowed-tools`: Read, Write, AskUserQuestion, Bash.

When a project completes:
1. Write the retrospective using the Project Retrospective template.
2. Move final diagrams to `completed/{name}/diagrams/` (if any).
3. **Remove `active/{name}/` folder** (its content is now in `completed/`).
4. **Keep `~/projects/{name}/`** (the code stays where it is).
5. Optionally update `02-ana/identity/projects.md` to reflect the project is done.

### Code lives outside the vault

```
~/projects/{name}/      ← Code. Outside the vault.
```

Why outside?
- The vault is for thinking, planning, and knowledge.
- Code has its own dependency tree (node_modules, target/, .venv) that doesn't belong in a thought-vault.
- Code wants its own git repo separate from the vault.
- Editor tooling (LSP, hot reload) works better when the code is in a focused project root.

The vault references the code via a link (`~/projects/{name}/` mentioned in `active/{name}/links.md` or `completed/{name}/links.md`). It does not contain the code. Completed retrospectives record the real path — e.g. Cortex at `~/projects/Cortex`, Helios at `~/work/helios`.

## Common questions

### Where do I put code?

`~/projects/{name}/`. Outside the vault.

### Where does the project's PRD live during development?

`05-projects/active/{name}/PRD.md`. Or `~/projects/{name}/docs/PRD.md` if you prefer it next to the code. The vault PRD is the durable record; the code-side PRD (if any) is for code-context convenience.

### What if a project is for a paid engagement?

Two valid placements:
- In `04-work/{engagement}/` if the primary frame is "paid engagement project."
- In `05-projects/active/{name}/` (or `completed/`) if the primary frame is "thing I am building."

A project can legitimately live in BOTH when it is both paid and personal — Helios is the canonical example (engagement folder + completed retrospective). That is intentional overlap, not duplication.

### When do I move from kitchen to active?

When you start writing code (or doing the work). Kitchen is "planning, no execution." Active is "execution underway." (No project is in `active/` right now — both current projects, open-kit and cloud-lab, are still in kitchen.)

### When do I move from active to completed?

When the project is shipped, abandoned, or otherwise done. There is no "in progress forever" — at some point an active project either completes or is killed. Helios went straight from its engagement build to `completed/` when the merge directive closed the standalone window.

### Can I have multiple active projects?

Yes. `05-projects/active/` can hold many. The constraint is your time, not the folder.

### What about a paused project?

Move it back to `kitchen/` or leave it in `active/` with a note in PRD.md that it is paused. There is no `paused/` folder. Status is in the document, not the folder. (This is exactly what happened to the homelab idea — it was reworked in kitchen into cloud-lab rather than abandoned.)

### What if a project completes and then I revive it?

Move it from `completed/` back to `active/`. Update the retrospective to note the revival. The project lifecycle is not strictly one-way.
